# 🏗️ 구조화 에이전트 상세 가이드

> 아이디어를 실행 가능한 구조로 정리
> 기능 정의, 요구사항 작성, 로드맵 수립

---

## 📋 개요

### 에이전트 목표
**다양한 아이디어를 우선순위화하고 실행 가능한 형태로 변환하여, 개발팀이 즉시 작업할 수 있는 명세서를 만듭니다.**

### 언제 사용하는가?

✅ **필수**:
- 아이데이션이 완료되고 아이디어 리스트가 있을 때
- 기능을 정의하고 요구사항을 문서화해야 할 때
- 개발 일정을 계획해야 할 때
- 팀 간(PM/개발/디자인) 정렬이 필요할 때

❌ **불필요**:
- 아이디어가 이미 명확하게 구조화되어 있을 때 → 바로 검증으로
- 개념 단계만 필요할 때 → 아이데이션으로

### 기대되는 결과

| 산출물 | 설명 | 용도 |
|--------|------|------|
| 기능 정의서 | 각 기능의 요구사항 상세 | 개발팀 기본 문서 |
| 사용자 스토리 | User Story 형식의 요구사항 | Agile 개발 스프린트 계획 |
| 로드맵 | Phase별 기능 배치 | 일정 및 리소스 계획 |
| 기획 명세서 | 최종 종합 문서 | 이해관계자 커뮤니케이션 |
| 복잡도/일정 | 기능별 소요 기간/리소스 | 프로젝트 관리 |

---

## 🎯 단계별 프로세스 (8 Steps)

### **Step 1: 평가 기준 및 우선순위 방식 정의** (1시간)

#### 목표
어떤 기준으로 아이디어를 평가하고 우선순위를 정할지 정의합니다.

#### 실행 방법

**1-1. MoSCoW 방식 선택 (추천)**
```
MoSCoW: 우선순위 분류

Must Have (필수)
- 없으면 제품이 작동하지 않음
- MVP에 반드시 포함
- 예: 칸반보드 기본 기능

Should Have (중요)
- 있으면 훨씬 좋음
- 초기 Phase에 포함 권장
- 예: 하위 작업 계층화

Could Have (있으면 좋음)
- Nice-to-have
- Phase 2-3에 포함
- 예: 테마 커스터마이징

Won't Have (제외)
- 현재 버전에서 제외
- 미래 검토 대상
- 예: VR 협업 인터페이스
```

**1-2. 평가 기준 정의**
```
각 기능을 평가할 기준:

1. 사용자 니즈 만족도 (1-5 점)
   - 사용자가 얼마나 원하나?

2. 비즈니스 가치 (1-5 점)
   - 매출/고객 확보에 기여도?

3. 기술적 실현성 (1-5 점)
   - 개발 가능한가?

4. 개발 복잡도 역방향 (1-5 점)
   - 5=간단, 1=복잡

5. 시장 시의성 (1-5 점)
   - 지금 꼭 필요한가? vs 나중에?

우선순위 계산:
점수 = (니즈×20% + 가치×30% + 실현성×20% + 복잡도역×20% + 시의성×10%)
```

**1-3. 우선순위 방식 선택**
```
방식 선택:

방식 A: MoSCoW (추천)
- 분류가 명확
- Agile 개발에 적합

방식 B: Kano Model
- 기본 요구사항 vs 매력 요소 구분
- 고급 분석

방식 C: Value vs 노력 매트릭스
- 시각적 명확성
- 팀 토론 용이
```

#### 체크리스트
- [ ] MoSCoW 또는 선택한 방식 결정
- [ ] 평가 기준 5-6개 정의
- [ ] 가중치 설정
- [ ] 팀 동의

---

### **Step 2: 아이디어 우선순위 결정** (2-3시간)

#### 목표
생성된 모든 아이디어를 MoSCoW 기준으로 분류합니다.

#### 실행 방법

**2-1. 아이디어별 평가**
```
평가표 작성:

아이디어명 | 니즈 | 가치 | 실현 | 복잡도 | 시의성 | 합계 | 분류
AI우선순위 | 5 | 5 | 4 | 4 | 5 | 4.6 | MUST
심플온보딩 | 5 | 4 | 5 | 5 | 4 | 4.6 | MUST
팀협업강화 | 4 | 4 | 4 | 4 | 3 | 3.9 | SHOULD
음성추가 | 3 | 3 | 2 | 3 | 2 | 2.6 | COULD
...
```

**2-2. 그룹별 분류**
```
분류 결과:

MUST HAVE (필수 기능):
- 칸반보드
- 할 일 추가/완료
- 팀원 초대
- AI 우선순위
- 심플 온보딩

SHOULD HAVE (중요 기능):
- 팀 협업 강화
- 실시간 동기화
- 모바일 앱
- 기본 리포팅
- 커스텀 필드

COULD HAVE (있으면 좋은):
- 음성 추가
- 자동화 규칙
- 고급 리포팅
- 테마 커스터마이징
- API 공개

WON'T HAVE (미포함):
- VR 인터페이스
- AI 코칭
- 가상 화이트보드
- 게임화
```

**2-3. MVP 범위 확정**
```
MVP (Minimum Viable Product) 정의:

MVP = 최소한으로 가치를 전달할 수 있는 기능 집합

MVP 범위:
├─ MUST HAVE 기능 (모두)
│  ├─ 칸반보드
│  ├─ AI 우선순위
│  ├─ 팀원 초대
│  └─ 심플 온보딩
│
└─ SHOULD HAVE 기능 (일부)
   └─ 팀 협협업 (필수)

MVP 미포함:
- 모바일 앱 (웹 먼저)
- 고급 리포팅 (기본만)
- 자동화 규칙 (후속)

MVP로 달성하는 가치:
"AI가 우선순위를 정해주는 심플한 협업 도구"
```

#### 체크리스트
- [ ] 모든 아이디어에 평가점수 배정
- [ ] MUST/SHOULD/COULD/WON'T 분류 완료
- [ ] MVP 범위 명확히 정의
- [ ] Phase별 기능 로드맵 초안

---

### **Step 3: 각 기능별 요구사항 정의** (4-6시간)

#### 목표
MVP에 포함된 각 기능의 상세 요구사항을 정의합니다.

#### 실행 방법

**3-1. 기능별 요구사항 템플릿**
```
기능명: AI 우선순위 정렬

1. 기능 개요
   - 정의: 팀의 모든 할 일을 중요도/긴급도 기반으로 자동 정렬
   - 핵심 가치: 사용자가 우선순위 고민 불필요
   - 영향도: 높음 (차별화 기능)

2. 기능 요구사항
   - 기능적 요구사항 (Functional)
     * 할 일 입력 시 AI가 자동으로 우선순위 제안
     * 사용자는 제안을 수락/거절 가능
     * 정렬 기준 설명 제시

   - 비기능적 요구사항 (Non-functional)
     * 응답 시간: 2초 이내
     * 정확도: 80% 이상
     * 가용성: 99.5% 이상

3. 사용자 인터페이스
   - 입력: 새 할 일 추가 시 자동 제안
   - 출력: 우선순위 배치 + 설명 (예: "프로젝트 마감일 임박")
   - 인터랙션: 드래그-드롭으로 수동 조정 가능

4. 비즈니스 규칙
   - 규칙1: 마감일 임박 작업은 상단
   - 규칙2: 팀의 의존 작업은 상단
   - 규칙3: 사용자 완료율 고려 (쉬운 것부터?)

5. 제약사항
   - 데이터: 개인정보보호법 준수
   - 성능: 100K 할 일 처리 가능
   - 호환성: 웹/모바일 동일 경험

6. 용어집 (Glossary)
   - AI 모델: 우리의 독점 알고리즘
   - 중요도: 비즈니스 가치
   - 긴급도: 시간 제약
```

**3-2. 모든 MVP 기능의 요구사항**
```
MVP 기능별 요구사항:

1. 칸반보드 (보드 → 리스트 → 카드)
   - 요구사항 5-10개
   - 복잡도: 중간

2. AI 우선순위 정렬
   - 요구사항 8-12개
   - 복잡도: 높음

3. 팀원 초대 및 권한
   - 요구사항 4-6개
   - 복잡도: 낮음

4. 심플 온보딩
   - 요구사항 6-8개
   - 복잡도: 중간

5. 팀 협업 (댓글, 멘션)
   - 요구사항 5-7개
   - 복잡도: 중간
```

**3-3. 요구사항 검증**
```
각 요구사항이 다음을 만족하는가?

S (Specific): 구체적인가?
  ❌ "빠르게 작동한다"
  ✅ "2초 이내에 응답한다"

M (Measurable): 측정 가능한가?
  ❌ "사용하기 좋다"
  ✅ "사용성 테스트에서 80% 만족도"

A (Achievable): 달성 가능한가?
  ✓ 일반적인 기술로 구현 가능

R (Relevant): 관련성 있는가?
  ✓ 비즈니스 목표와 연결

T (Time-bound): 기한이 있는가?
  ✓ MVP 범위 내에 포함 (3개월 내)
```

#### 체크리스트
- [ ] 각 기능별 상세 요구사항 작성 (15-20개)
- [ ] 기능적/비기능적 요구사항 분리
- [ ] 용어집 작성
- [ ] SMART 검증 완료

### **Step 4: 사용자 스토리 작성** (2-3시간)

#### 목표
각 기능을 사용자 관점의 "스토리" 형태로 작성합니다.

#### 실행 방법

**4-1. 사용자 스토리 포맷**
```
기본 포맷:
"As a [사용자 역할], I want to [사용자가 원하는 것],
 so that [그 이유/목표]"

예시:
"As a 팀 리더, I want to AI가 우선순위를 정해주길 원합니다,
 so that 팀의 생산성을 높이고 회의 시간을 줄일 수 있습니다"
```

**4-2. 사용자 스토리 작성**
```
Feature: AI 우선순위 정렬

Story 1:
As a 팀 리더
I want to 새 할 일이 추가될 때 자동으로 우선순위를 받고 싶습니다
So that 팀의 목표와 일정에 맞는 순서대로 작업할 수 있습니다

Acceptance Criteria:
- [ ] 할 일 추가 시 우선순위 제안 표시
- [ ] 제안 근거 설명 제시 (예: "마감일 임박")
- [ ] 사용자가 제안 수락/거절 가능
- [ ] 수동 조정도 가능

Story 2:
As a 개별 팀원
I want to 내 할 일 리스트가 자동으로 정렬되길 원합니다
So that 뭘 먼저 해야 할지 고민하지 않아도 됩니다

Acceptance Criteria:
- [ ] 개인 태스크도 AI 정렬 적용
- [ ] 정렬 기준 명확 (왜 이 순서?)
- [ ] 성능 영향 최소화

Story 3:
As a 경영진
I want to 팀의 생산성 개선을 추적하고 싶습니다
So that AI 기능의 효과를 검증할 수 있습니다

Acceptance Criteria:
- [ ] 우선순위 정확도 대시보드
- [ ] 생산성 개선율 표시
- [ ] 월간 리포트 자동 생성
```

**4-3. 스토리 포인트 (복잡도) 평가**
```
포인트 시스템 (Fibonacci):
1점: 극히 간단 (1시간 이내)
2점: 매우 간단 (반나절)
3점: 간단 (1일)
5점: 중간 (2-3일)
8점: 복잡 (1주)
13점: 매우 복잡 (1주+)
21점: 극도로 복잡 (분해 필요)

각 스토리 포인트 배정:

Story 1 (우선순위 제안): 8점
Story 2 (개인 태스크): 5점
Story 3 (생산성 대시보드): 13점
```

#### 체크리스트
- [ ] MVP 기능별 사용자 스토리 3-5개씩 작성
- [ ] 각 스토리에 수락 조건(Acceptance Criteria) 명시
- [ ] 스토리 포인트 평가 완료
- [ ] 최소 15-20개 스토리 작성

### **Step 5: 기능별 복잡도/영향도 평가** (2-3시간)

#### 목표
각 기능의 개발 난이도와 비즈니스 영향도를 정량화합니다.

#### 실행 방법

**5-1. 복잡도 평가**
```
복잡도 평가 기준:

기술 복잡도 (1-5):
1 = CRUD 기본 기능 (DB 저장/조회)
2 = 기존 기능 확장
3 = 새로운 기술 필요 (검색, 캐싱)
4 = 복잡한 알고리즘 필요
5 = 혁신적 기술/연구 필요

설계 복잡도 (1-5):
1 = 명확한 디자인 기존
2 = 약간의 UX 설계 필요
3 = 새로운 UX 패턴
4 = 복잡한 인터랙션
5 = 새로운 UI 패러다임

총 복잡도 = (기술 + 설계) / 2
```

**5-2. 비즈니스 영향도 평가**
```
영향도 평가:

사용자 가치 (1-5):
- 사용자에게 얼마나 중요한가?

차별화 효과 (1-5):
- 경쟁사와의 차별화?

고객 획득 임팩트 (1-5):
- 신규 고객 확보에 도움?

매출 기여 (1-5):
- 수익화 기여도?

총 영향도 = (가치 + 차별화 + 획득 + 매출) / 4
```

**5-3. 영향도 vs 복잡도 매트릭스**
```
시각화:

        ▲ 영향도
        │
   높음 │  ●●● (AI우선순위, 온보딩)
        │ ●  ● (협업, 동기화)
   중간 │
        │ ●●  (댓글, 권한)
   낮음 │    ●●●●
        └─────────────────→ 복잡도
           쉬움   중간   어려움

우선순위:
1순위: 높은 영향도 + 낮은 복잡도 (오른쪽 위)
2순위: 높은 영향도 + 중간 복잡도
3순위: 중간 영향도 + 낮은 복잡도
나중: 낮은 영향도 + 높은 복잡도
```

#### 체크리스트
- [ ] 모든 기능의 기술 복잡도 평가
- [ ] 모든 기능의 설계 복잡도 평가
- [ ] 비즈니스 영향도 평가
- [ ] 영향도 vs 복잡도 매트릭스 시각화

---

### **Step 6: 개발 일정 예상 및 리소스 계획** (1-2시간)

#### 목표
각 기능의 예상 개발 기간과 필요 리소스를 파악합니다.

#### 실행 방법

**6-1. 기능별 소요 기간 추정**
```
추정 방법: 스토리 포인트 기반

팀의 Velocity (주당 처리 가능 포인트):
- 경험 많은 팀: 40-50 포인트/주
- 중간 팀: 20-30 포인트/주
- 신입 팀: 10-15 포인트/주

예시 (주당 25포인트):

Feature 1: 칸반보드
- 스토리 1 (8pt) + 스토리 2 (5pt) + 스토리 3 (3pt) = 16pt
- 버퍼 (20% 증가) = 19.2pt ≈ 1주

Feature 2: AI 우선순위
```

## ❓ Q&A

### Q1: MUST/SHOULD/COULD를 어떻게 나누나요?
**A**: 기준
- MUST: 없으면 제품이 아님 (MVP에 필수)
- SHOULD: 있으면 훨씬 좋음 (경쟁에서 필요)
- COULD: Nice-to-have (미래에)

의심될 때는 SHOULD로 분류

### Q2: 사용자 스토리는 정말 필요한가요?
**A**: 네, 매우 중요합니다
- 개발팀이 "무엇을 만들지" 이해 가능
- 테스트 기준이 명확 (Acceptance Criteria)
- Agile 개발의 핵심

### Q3: 일정 추정이 자주 빗나가면?
**A**:
- 버퍼 추가 (20-30%)
- 팀의 Velocity 기반 재계산
- 정기 리뷰 (주 1회)
