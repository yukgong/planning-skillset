# 사용자중심 에이전트 (User-Centered Agent) 가이드

## 📋 개요

사용자중심 에이전트는 기획된 서비스나 기능이 실제 사용자의 요구사항을 얼마나 잘 충족하는지를 검증합니다. 페르소나 정의부터 시작하여 사용자 여정 맵, 와이어프레임, 프로토타입, 그리고 사용성 테스트를 통해 사용자 경험을 최적화합니다.

### 🎯 목표
- 타겟 사용자의 명확한 이해 (페르소나, 니즈, 행동)
- 사용자 여정의 모든 접점(Touchpoint) 설계
- 직관적이고 매력적인 사용자 인터페이스 구현
- 실제 사용자 피드백을 통한 검증
- 최종적으로 사용성과 만족도 극대화

### 💡 언제 사용하는가?
- 기능이나 서비스 기획이 완료되었을 때
- 사용자 경험의 질을 높이고 싶을 때
- 기존 고객과의 갭을 파악하고 싶을 때
- 출시 전 사용성을 검증하고 싶을 때
- 사용자 만족도를 최우선으로 하고 싶을 때

---

## 🎯 단계별 프로세스

### 📌 Step 1: 사용자 페르소나 정의 (2-3시간)

**목표:** 타겟 사용자를 명확하게 정의하고 대표 페르소나를 만듭니다.

#### 실행 방법

1. **고객 세그멘테이션**
   - 고객군별 분류 (기업규모, 산업, 직급)
   - 각 세그먼트의 특성 파악
   - 우선순위 결정

2. **페르소나 작성**
   - 이름, 나이, 직급 등 기본 정보
   - 업무 책임, 목표, 도구 사용 경험
   - 통증점(Pain Point), 동기부여 요소
   - 의사결정 프로세스

3. **페르소나별 우선순위**
   - 1차 페르소나 (Primary): 주요 타겟
   - 2차 페르소나 (Secondary): 중요 타겟
   - 3차 페르소나 (Tertiary): 보조 타겟

4. **검증**
   - 사용자 인터뷰로 페르소나 검증 (5-10명)
   - 고객 데이터 분석으로 재검증
   - 필요시 페르소나 조정

#### 페르소나 템플릿

```markdown
## 페르소나 1: 마케팅 매니저 (1차 페르소나)

### 기본 정보
- 이름: 김지은
- 나이: 32세
- 직급: 마케팅 매니저
- 회사: 스타트업 (50-100명)
- 경력: 5년

### 배경
- 마케팅 팀 3명 관리
- 월 10회 이상 팀 회의 진행
- 회의 내용 기록 및 공유 담당
- 영업팀과 월 1회 협업 회의

### 업무 책임
- 월간 마케팅 캠페인 기획 및 실행
- 팀 회의 진행 및 기록
- 임원진에 결과 보고
- 회의 기록 보관 및 추후 참조

### 사용 도구
- Slack (주요 협업 도구)
- Google Workspace
- Notion (회의록 저장)
- Excel (분석 및 리포팅)

### 통증점 (Pain Points)
- 회의 중 메모 작성으로 집중력 흐름
- 회의 후 기록 작성에 30분 이상 소비
- 주요 액션 아이템 놓치는 경우 발생
- 팀원들이 회의 내용을 다시 물어봄
- 임원진 리포트 작성에 시간 소비

### 동기부여 요소
- 회의 시간 단축 → 실제 업무 시간 확보
- 자동 기록으로 생산성 향상
- 팀원 간 정보 공유 자동화
- 회의 인사이트 분석

### 기술 숙련도
- SaaS 도구 사용 경험: 높음
- 새로운 도구 학습: 빠름
- 자동화 기능 활용: 중간

### 의사결정 프로세스
- 무료 트라이얼: 필수
- 팀 의견 수렴: 중요
- ROI 검증: 필수
- 의사결정 기간: 1개월

### 예상 사용 시나리오
1. 주 1회 팀 회의 예약
2. 회의 중 AI 요약 기능 활용
3. 회의 후 자동 생성된 요약본 검토
4. 팀 슬랙에 요약 자동 공유
5. 월말 임원진 리포트에 회의 내용 재사용
```

#### 다양한 페르소나 예시

```
페르소나 2: 영업 담당자 (2차 페르소나)
- 이름: 이준호
- 나이: 28세
- 직급: 영업 담당자
- 회사: 중견기업 (200명)
- 주요 니즈: 고객 미팅 기록, 따라잡기 회의 자동화
- 통증점: 미팅 후 고객사에 회의록 보내야 함, 팀에 진행상황 공유 번거로움

페르소나 3: IT 관리자 (3차 페르소나)
- 이름: 박철수
- 나이: 42세
- 직급: IT 인프라 담당자
- 회사: 대기업 (1,000명)
- 주요 니즈: 보안, 컴플라이언스, 기술 지원
- 통증점: 데이터 보안, 레거시 시스템 통합
```

### 📌 Step 2: 사용자 여정맵 작성 (3-4시간)

**목표:** 사용자가 제품을 인지부터 사용 후까지의 전체 여정을 시각화합니다.

#### 실행 방법

1. **여정 단계 정의**
   - Awareness (인식): 제품을 처음 알게 됨
   - Consideration (고려): 제품 정보 수집
   - Purchase (구매): 구매 결정 및 실행
   - Onboarding (온보딩): 제품 처음 사용
   - Usage (사용): 일상 업무 활용
   - Advocacy (옹호): 다른 사람에게 추천

2. **각 단계별 사용자 행동 정의**
   - 사용자가 취하는 구체적 행동
   - 사용하는 도구/채널
   - 느끼는 감정 (긍정/부정)

3. **접점(Touchpoint) 식별**
   - 회사(우리)와 사용자가 상호작용하는 지점
   - 각 접점에서의 커뮤니케이션
   - 개선 기회

4. **문제점 식별 및 개선안 제시**
   - 각 단계에서의 장애물
   - 사용자 불만 요소
   - 개선 방안

#### 사용자 여정맵 예시

```markdown
## 사용자 여정맵: 협업도구 SaaS 회의 요약 기능

### Stage 1: 인식 (Awareness)
**사용자 행동**
- LinkedIn에서 "AI 회의 기록" 광고 본다
- ProductHunt에서 신제품 리뷰 본다
- 동료로부터 제품 추천받는다

**감정**
😐 → 🤔 (호기심 생성)

**우리의 접점**
- 광고 (LinkedIn, Google)
- 제품 리뷰 사이트
- 입소문 (마케팅)

**문제점**
- 광고 클릭율 낮음
- 제품 설명 복잡함

**개선안**
- 간단한 한 줄 설명 강조
- 동영상 데모 제공
- 무료 트라이얼 강조

---

### Stage 2: 고려 (Consideration)
**사용자 행동**
- 웹사이트 방문
- 동영상 튜토리얼 시청
- 가격표 확인
- 경쟁사 제품 비교

**감정**
🤔 → 😊 (호기심 증가)

**우리의 접점**
- 웹사이트 랜딩페이지
- 비디오 튜토리얼
- 가격 정보
- 비교 차트

**문제점**
- 랜딩페이지에 너무 많은 정보
- 가격 정보 불명확
- 경쟁사 비교 어려움

**개선안**
- 랜딩페이지 단순화 (3개 핵심 기능만)
- 명확한 가격표
- 비교 테이블 제공

---

### Stage 3: 구매 (Purchase)
**사용자 행동**
- 무료 트라이얼 시작
- 신용카드 등록
- 구독 플랜 선택

**감정**
😊 → 😟 (약간의 불안감)

**우리의 접점**
- 회원가입 프로세스
- 결제 페이지
- 확인 이메일

**문제점**
- 회원가입 필드 많음 (10개 이상)
- 결제 오류 시 고객지원 응답 느림
- 구독 시작 후 가이드 없음

**개선안**
- 회원가입 필드 3개로 단순화
- 결제 오류 자동 재시도
- 구독 완료 후 온보딩 이메일 자동 발송

---

### Stage 4: 온보딩 (Onboarding)
**사용자 행동**
- 제품 처음 로그인
- 기본 설정 (계정, 권한)
- 첫 회의 요약 시도

**감정**
😟 → 😕 (초반 혼란) → 😊 (점진적 이해)

**우리의 접점**
- 온보딩 튜토리얼
- 앱 내 가이드
- 고객지원 챗봇
- 이메일 가이드

**문제점**
- 튜토리얼이 너무 길다 (15분 이상)
- 실제 회의 연동 복잡함
- 첫 사용 시 오류 발생

**개선안**
- 튜토리얼 3분으로 단축
- 1-클릭 Slack 연동
- 오류 시 즉시 도움말 표시

---

### Stage 5: 사용 (Usage)
**사용자 행동**
- 주 1회 회의에 사용
- 팀과 요약 공유
- 요약본 검수 및 수정
- 피드백 제공

**감정**
😊 → 😍 (만족감 증가)

**우리의 접점**
- 모바일/웹 앱
- Slack 통합
- 이메일 공유
- 고객 피드백 채널

**문제점**
- 정확도 이슈 (93%)
- 기능 요청 반영 느림
- 기술지원 응답 시간 길다

**개선안**
- 정확도 개선 (95% 이상 목표)
- 월간 기능 업데이트 명확화
- 24시간 기술지원

---

### Stage 6: 옹호 (Advocacy)
**사용자 행동**
- 팀원에게 추천
- 사내 다른 팀에 공유
- 리뷰 사이트에 리뷰 작성
- 컨퍼런스에서 사례 공유

**감정**
😍 → 😄 (강한 만족감)

**우리의 접점**
- 추천 프로그램
- 사례 공개
- 고객 커뮤니티
- 파트너 네트워크

**문제점**
- 추천 인센티브 불명확
- 사례 공개 절차 복잡
- 커뮤니티 활동도 낮음

**개선안**
- 명확한 추천 리워드 ($100/건)
- 사례 작성 템플릿 제공
- 월간 고객 웨비나 개최
```

#### 여정맵 시각화 요소

```
여정 단계별 감정 그래프:

감정도
5  ╱╲
4 ╱  ╲╱─────╲
3        ╱   ╲╱
2       ╱
1

   Awareness → Consideration → Purchase → Onboarding → Usage → Advocacy
```

### 📌 Step 3: 사용자 니즈 및 기대사항 분석 (2-3시간)

**목표:** 각 단계에서 사용자가 원하는 것과 기대하는 것을 명확히 파악합니다.

#### 실행 방법

1. **기능적 니즈 (Functional Needs)**
   - 사용자가 해결하고 싶은 업무 문제
   - 필요한 기능 (핵심 vs 부가)
   - 성능 요구사항

2. **감정적 니즈 (Emotional Needs)**
   - 사용자가 느끼고 싶은 감정
   - 신뢰감, 자신감, 편안함
   - 사회적 인정

3. **사회적 니즈 (Social Needs)**
   - 팀 협업 요구사항
   - 정보 공유 방식
   - 사회적 지위 표현

#### 니즈 분석 매트릭스

```markdown
## 사용자 니즈 분석

### 마케팅 매니저 (김지은)

#### 기능적 니즈
| 니즈 | 중요도 | 기대 수준 | 우리 제공 수준 | 갭 |
|------|--------|---------|------------|-----|
| 자동 음성인식 | 높음 | 95% 정확도 | 94% | -1% ⚠️ |
| 액션 아이템 추출 | 높음 | 완벽 | 85% | -15% ⚠️ |
| 요약본 자동 생성 | 높음 | 3초 이내 | 2초 | +1초 ✓ |
| 다중 언어 지원 | 중간 | 한/영 | 한/영/일 | ✓ |
| Slack 통합 | 높음 | 원클릭 | 2클릭 | -1클릭 ⚠️ |
| 검색 기능 | 중간 | 빠름 | 1초 | ✓ |

#### 감정적 니즈
- 신뢰감: "AI가 중요한 내용을 놓치지 않을까?" (신뢰도 강화 필요)
- 편안함: "간단하게 써도 될까?" (사용 난이도 낮춤)
- 자신감: "내 업무 효율이 정말 올라갈까?" (성과 입증)

#### 사회적 니즈
- 팀원들과의 정보 공유 (Slack 자동 공유 ✓)
- 임원진에 진행상황 보고 (내보내기 기능 필요)
- 업계 커뮤니티 공유 (케이스 스터디 ✓)
```

### 📌 Step 4: 와이어프레임 및 프로토타입 작성 (4-6시간)

**목표:** 사용자 인터페이스의 구조와 흐름을 시각화합니다.

#### 실행 방법

1. **정보 구조 (IA) 설계**
   - 주요 화면 정의 (5-10개)
   - 화면 간 네비게이션
   - 정보 계층(hierarchy) 정의

2. **와이어프레임 작성**
   - 저화질(Low-fidelity) 와이어프레임
   - 버튼, 입력 필드, 텍스트 배치
   - 사용자 흐름 표시

3. **프로토타입 개발**
   - 고화질(High-fidelity) 프로토타입
   - 실제 디자인과 색상
   - 상호작용 정의 (클릭, 드래그 등)
   - 애니메이션 및 피드백

4. **사용성 테스트 준비**
   - 테스트 시나리오 작성
   - 관찰 포인트 정의
   - 질문 항목 준비

#### 와이어프레임 예시

```markdown
## 화면 1: 회의 요약 대시보드

┌─────────────────────────────────────────┐
│ 🔙 회의 요약 | 🔔 | 👤 | ⚙️            │
├─────────────────────────────────────────┤
│                                         │
│  이번 주 회의 3건                      │
│  ┌──────────────────────────────────┐  │
│  │ 📅 2024-01-15                   │  │
│  │ 마케팅 팀 회의                   │  │
│  │ 🎤 18분 37초 | 👥 5명          │  │
│  │ 📊 요약본: [요약 본문...]        │  │
│  │ ✅ 액션 아이템 3개               │  │
│  │ [복사] [공유] [편집]            │  │
│  └──────────────────────────────────┘  │
│                                         │
│  ┌──────────────────────────────────┐  │
│  │ 📅 2024-01-12                   │  │
│  │ 임원진 회의                      │  │
│  │ ...                             │  │
```

## ❓ Q&A

### Q1: 페르소나는 몇 개가 필요한가요?

**A:** 일반적으로 3-5개입니다.
- Primary (1개): 가장 중요한 사용자
- Secondary (1-2개): 중요한 사용자
- Tertiary (1-2개): 보조 사용자

참고: 너무 많으면 설계 결정이 어려워집니다.

### Q2: 사용성 테스트에 몇 명이 필요한가요?

**A:** 최소 5명, 권장 8-10명입니다.
- 5명: 전체 문제의 85% 발견
- 8-10명: 전체 문제의 95% 이상 발견
- 15명 이상: 중복 발견 증가

### Q3: 우리 제품이 복잡하면 어떻게 하나요?

**A:** 단계적 테스트를 권합니다:
1. 핵심 기능만 테스트 (30분)
2. 고급 기능 테스트 (별도 테스트)
3. 통합 시나리오 테스트

### Q4: 테스트 후 설계 변경이 커도 괜찮나요?

**A:** 네, 정상입니다. 재테스트를 진행합니다:
- 큰 변경 (> 30% UI 변경): 재테스트 필수 (3-5명)
- 작은 변경 (< 30%): 내부 검증으로 충분

### Q5: 원격 테스트와 대면 테스트 중 어느 것이 낫나요?

**A:** 둘 다 유효합니다:
- 대면: 비언어적 신호 관찰 가능, 높은 참여도
- 원격: 비용 저렴, 지역 제약 없음, 시간 효율

권장: 1차는 원격, 중요한 개선 후 대면 테스트

- [ ] 6개 단계 정의 (Awareness → Advocacy)

- [ ] 감정 곡선 표시
- [ ] 접점 식별
- [ ] 문제점 식별
- [ ] 개선안 제시

### 니즈 분석 (Step 3)

- [ ] 감정적 니즈 파악
- [ ] 사회적 니즈 확인
- [ ] 니즈-제공 갭 분석

### 와이어프레임 (Step 4)
- [ ] 주요 화면 5개 이상
- [ ] 화면 간 네비게이션

### 프로토타입 (Step 5)
